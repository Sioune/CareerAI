---
name: CareerAI admin back-office scope decision
description: Phase-1 scope chosen for the admin module against a much larger enterprise PRD.
---

A PRD (`CareerAI_后台管理系统_PRD_v1.0`) specifies a full enterprise admin system: 8-role RBAC, MFA, audit logs, versioned prices/prompts/config, append-only entitlement/finance/cost ledgers, revenue allocation, CMS, AI provider/prompt management — an 8-week team build.

**Why:** Given the app's actual scale (single Express+React app, one dev), building the full PRD in one pass isn't realistic. User agreed to a phase-1 subset first.

**How to apply:** Phase-1 admin module (`/admin`, `src/admin/AdminApp.tsx`, admin API under `/api/admin/*` in `server.ts`) covers: business overview dashboard, user list/detail, task list, payment list, real refunds via the existing ICBC gateway client (`refunds` table, `createRefund`/`queryRefundStatus`), and referral ledger view.

Phase-2 (added later) layered in a subset of the deferred scope: simple RBAC (`admins.role` ∈ super_admin/operations/finance/customer_service/auditor, `requireRole()` middleware in server.ts, super_admin bypasses all checks), an append-only `audit_logs` table logging login/refund/role-change events, and a `cost_events` table that estimates Gemini token cost per AI call (illustrative unit pricing, not real billing) to compute gross margin on the finance tab.

Phase-3 added Maker-Checker (dual-approval) refunds: requesting a refund only creates a pending `refunds` row (status=0) with no gateway call; a separate admin must call the approve endpoint, which then invokes the real ICBC `createRefund`. Self-approval is blocked unless the approver is `super_admin` (later also applied to rejection — see Phase-5).

Phase-4 (full backend + frontend, both verified working via curl and in AdminApp.tsx) closed most remaining PRD gaps: MFA (TOTP via `speakeasy`, NOT `otplib` — otplib's v13 API lacks the `authenticator` import used in older docs, caused a real bug), Site Config/CMS with draft→publish versioning, AI Providers/Models/Prompt-Versions management, Support Tickets (public create endpoint is `/api/admin/tickets` POST with no auth — there is no separate `/api/tickets`), Notifications (admin compose + public feed), and Risk Control (rule evaluation wired into payment success and refund approval, with a review/dismiss admin UI).

Still deferred (schema tables exist, no endpoints/UI): JD data source management, evaluation/experiments module, revenue-allocation ledger logic (`revenueAllocations` table unused). Also still open: config versioning only supports publishing the latest draft, not rolling back to an arbitrary older version; AI provider/model *creation* has no form (API-only, list+toggle only in UI).

Phase-5 (PRD-alignment rework, structural only) introduced a single RBAC source of truth at `src/shared/permissions.ts` imported by BOTH `server.ts` and `src/admin/AdminApp.tsx` so gating can never drift: 8 roles (PRD §2.1), 14 permission modules, a none/read/write MATRIX (PRD §2.3), and `hasPermission(role,module,action)` (super_admin always full). Server enforces via a `requirePermission(module,action)` middleware on every mutating/reading `/api/admin/*` route (self-service login/`/me`/MFA and the gateway `/refunds/callback` stay ungated; refund approve/reject additionally keep `requireRole("finance")`). Added an approval center: `approvals` table + `GET /api/admin/approvals` (operations/customer_service see only own-initiated; finance/auditor/super see all) and `POST /approvals/:id/approve|reject`; refund-type approvals delegate to the same guarded `approveRefundById`/`rejectRefundById` helpers. The admin UI was rebuilt to the PRD §3.1 nine-module two-level nav, filtered by `hasPermission`, with an `ApprovalsTab` and "规划中" placeholder pages for deferred Phase-2/3 modules. Phase 2/3 (real product/price/config/prompt publish approvals, ledgers, CMS, reconciliation) still deferred — get user sign-off before starting.

**Maker-Checker invariant (learned in Phase-5):** the self-approval block must be enforced *server-side in every terminal path*, not just hidden in the UI. It now lives in `approveRefundById`, `rejectRefundById`, AND the generic (non-refund) branches of both `/approvals/:id/approve` and `/approvals/:id/reject` — the UI's `canActOn`/button-hiding is defense-in-depth only. **Why:** an earlier version blocked self-approval only for refund-approve and in the UI, so a finance admin could self-approve/reject non-refund approvals via direct API call; UI and server had silently drifted on the exact invariant the feature exists to guarantee.

Phase 2A (publish-approval closed loop + product/price) made site-config / AI-prompt / price publishing go through the approval center instead of publishing directly: a `*/publish` endpoint now flips the version draft→pending and creates a `config_publish`/`prompt_publish`/`price_publish` approval; approving runs the shared `publish*Version` helper (archives all published siblings in the same key/operation/skuId group, then publishes the target); rejecting reverts pending→draft. Added `products`/`skus`/`priceVersions` tables and `payments.priceVersionId`+`priceSnapshot` (order creation snapshots the currently-published price matched by active SKU `targetRole`; snapshot is decoupled from the charged amount).

**Publish-approval bypass invariant (learned in Phase 2A):** version *rollback* endpoints (`config|ai/prompts|prices/:id/rollback`) call the same `publish*Version` helper the approval path uses, but the helper does NOT check the target's current status. So a rollback endpoint must itself require `status === "archived"` before calling the helper. **Why:** without that guard, any admin with mere module-write (operations/content/ai_ops) could create a *draft* and immediately publish it via the rollback route — no approval, no second person — silently defeating the whole maker-checker control on a live-pricing/config path. **How to apply:** any new endpoint that calls a `publish*Version` helper outside the approval dispatcher must gate on the source status (archived for rollback), never trust the id alone.

**Phase-2 permission simplification:** all `*_publish` approvals are gated on `approvals:write` (finance/super only), not a per-module approver. This is coherent because finance lacks products/site/ai write, so requester and approver roles are naturally disjoint (super_admin is the intentional bypass). If a future phase gives one non-super role both a module's write AND approvals:write, revisit — the maker-checker self-block would then be the only thing preventing self-approval.

Known separate long-standing issue (not part of these phases): `RESUME_UNLOCK_PRICE_CENTS` in server.ts and the matching frontend price constant are still a ¥0.01 test price, not the real ¥29.90 — flagged for a future session to fix.
